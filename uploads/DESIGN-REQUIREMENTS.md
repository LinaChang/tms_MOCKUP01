## 1. 專案定位

- 產品名稱:行銷部任務與內容管理系統(mkt-tms)。
- 使用情境:行銷部內部任務進度追蹤、需求單管理、內容資源管理。
- 第一版定位:內部 live demo / 第一版範圍確認版。
- 使用者:行銷處、數位處、餐飲部的內部人員;完整館別與單位窗口帳號延後。
- 對外公開:第一版不對外公開,以登入牆保護;有網址的人只會看到登入頁。
- 設計目標:讓需求單位快速理解完整平台樣貌、工作流程、權限差異與資料關係。

## 2. 設計原則

- 這是內部管理工具,不是行銷官網;介面要安靜、清楚、可掃描、適合反覆操作。
- 優先設計高資訊密度的表格、篩選、狀態、列表與詳情頁。
- 避免過度卡片化、AI 風格、裝飾性 hero、漸層裝飾或大型情境插圖。
- 元件狀態要完整:預設、hover、focus、disabled、loading、empty、error、success。
- 所有可點擊項目都要有明確 affordance,例如需求單編號、需求單標題、專案名稱。
- 未完成流程在 live demo 可停住,但畫面上要清楚看得出目前流程位置與不可操作狀態。
- 介面文字第一版以繁體中文為主;英文只保留必要代碼、帳號、技術名詞。
- 禁用 emoji 與跨平台易亂碼符號;圖示一律使用 Lucide SVG。

## 5. 角色與權限的介面差異

系統共 4 種角色:

| 角色 | 介面能力 |
| ---- | -------- |
| 最高系統管理者 | 看所有資料;可新增、編輯、刪除、審核、上傳附件、管理設定 |
| 管理者 | 看所有資料;可新增、編輯、刪除、審核、上傳附件 |
| 編輯者 | 只能看與操作自己是 PM、協作或開單人的資料;可新增、編輯、上傳附件;不可刪除、不可審核、不可看所有資料 |
| 瀏覽者 | 只能瀏覽被設定公開的內容;不可新增、編輯、刪除、審核、上傳附件 |

設計要求:

- 同一頁需考慮不同角色看到的按鈕差異。
- 不可操作的按鈕不能只隱藏;若流程需要教育使用者,可用 disabled 狀態與短提示。
- 列表需支援「全部資料」與「我的資料」兩種權限視角。
- 瀏覽者需有明確的唯讀狀態。

## 7. 登入與系統入口

### 已決定

- 全系統只有一個入口。
- 不做免登入需求單開立入口。
- 不做獨立進度查詢入口。
- 登入後依角色與授權顯示可看或可編輯的功能。
- live demo 登入頁先提供 demo 帳號 / 角色按鈕,點擊後自動登入。
- 第一版不加 IP 限制或 Cloudflare Access。

### 登入頁設計需求

- 清楚標示系統名稱。
- 顯示 demo 帳號按鈕,按鈕需包含帳號、角色、部門。
- 可呈現正式登入欄位的版型,但 live demo 可先以 demo 按鈕為主要操作。
- 忘記密碼第一版不做使用者自行 email 重設;正式版若導入,另行設計。
- 需要錯誤、loading、disabled 狀態。

## 9. 儀表板

儀表板需協助使用者快速掌握目前工作狀態。

### 必要區塊

- 全部 / 我的需求單統計

- 個人或團隊工作負載摘要
- 即將到期專案

- 工作負載認定為 PM + 協作都算。

## 10. 需求單管理

需求單是第一版核心畫面,美工 / 設計需求與拍照需求都併入需求單。

### 列表頁

列表需支援:

- 需求單編號
- 專案名稱
- 館別
- 開單單位
- 專案類型
- 負責 PM
- 協作人員
- 開單人
- 開單日期
- 狀態
- 進度
- 需求大類
- 美工 / 設計製作分類
- 設計來源

建議篩選:

- 狀態
- 館別
- 處 / 部
- 專案類型
- PM
- 協作人員
- 需求大類
- 設計來源
- 日期區間

互動:

- 需求單編號與專案名稱需是可點擊連結。
- 需有「新增需求單」主按鈕。
- 新增需求單已決定拆成獨立頁面,由列表頁按鈕進入。

### 新增 / 編輯需求單

表單欄位:

- 需求單編號,系統自動產生
- 專案名稱
- 館別
- 開單單位
- 專案類型
- 負責 PM
- 協作人員,可多選
- 開單人
- 開單日期
- 狀態
- 進度,0-100
- 備註
- 額外項目,例如製作物輸出、社群媒體、EDM
- 需求大類,例如一般需求、美工設計、拍照需求
- 美工 / 設計製作分類
- 設計來源
- 退回 / 版本紀錄

設計要求:

- 欄位需分群,避免一整頁長表單難以掃描。
- 重要欄位需有必填狀態。
- 狀態、館別、專案類型、設計來源需使用下拉或分段選項。
- PM 與協作人員需用人員選擇器。
- 儲存、取消、送出、退回、完成需有明確位置。

### 詳情頁

詳情頁需呈現:

- 需求單基本資料
- 目前狀態與進度
- PM 與協作人員
- 執行項目
- 附件
- 簽核
- 審閱 / 溝通紀錄
- 退回 / 版本紀錄
- 採購單關聯

## 11. 美工 / 設計需求流程

美工單不獨立成新入口,而是需求單的一種。

### 分類

- 海報類
- 電子製作物,館內電子看板
- 菜單類
- 文宣印刷品
- 包裝類
- 場地布置類
- 特殊設計 / 其他

### 設計來源

- 館內設計
- 外包設計
- 待確認

### 流程呈現

流程建議以 stepper 或狀態軌呈現:

1. 開單單位提出需求
2. 行銷 / 公關初審
3. 指派 PM 與設計來源
4. 設計完成後回傳行銷 / 公關
5. 直屬主管確認
6. 相關單位確認
7. 完成結案

退回時不重新開單,需保留同一需求單號,用版本與審閱 / 溝通紀錄追蹤。

## 12. 拍照需求

拍照需求不做獨立拍照單,併入同一張需求單下的「拍照」執行項目。

畫面需支援:

- 拍攝日期
- 地點
- 主題
- 需求單位窗口
- 照片用途
- 交付格式
- 場景清單,待確認
- 修圖需求,待確認

拍照需求表單細項仍需需求單位確認。

執行項目需掛在需求單或專案下。

欄位:

- 任務名稱
- 狀態
- 負責人
- 協作
- 時程 / 截止日
- 說明
- 是否需報告
- 子任務
- 附件
- 任務類別,例如設計製作、拍照、外包溝通、主管確認

設計要求:

- 可用表格或分組列表呈現。
- 任務狀態需清楚。

## 14. 專案管理與進度追蹤

畫面需支援:

- PM 指派
- 協作人員管理
- 執行項目編輯
- 狀態變更
- 進度百分比
- 甘特圖入口
- 附件與溝通紀錄

設計要求:

- 專案與需求單關係要清楚。
- 狀態變更需有明確操作點。
- 權限不足者需看到唯讀或不可操作狀態。

## 15. 工作負載甘特圖

甘特圖是第一版必要重點。

### 已決定

- 每個人都要有專案甘特圖。
- 顯示層級先到專案,不展開到任務或子任務。
- 月報、季報、年報三種區間,用 tab 切換。
- 切到季報與年報時,以個人為主要檢視單位。
- 需要里程碑。
- 需要呈現任務相依或關聯關係。
- 工作負載認定:PM + 協作都算。

### 視覺設計要求

- PM 與協作需用不同顏色或標記區分。
- 主要專案與相依項目需區分。
- 里程碑需有明確符號,但不能使用 emoji。
- 同一人多專案重疊時,需能分層或堆疊顯示。
- 年度視圖資料較密,需避免文字互相覆蓋。
- 需提供圖例。
- 需有空狀態,例如該期間無專案。

介面需有:

- 上傳區
- 檔案類型標示
- 檔案大小
- 上傳進度
- 上傳成功 / 失敗狀態
- 權限不足不可下載狀態

欄位:

- 名稱
- 連結網址

- 說明

## 32. 響應式與無障礙

- 第一版只支援桌機與平板橫向;最小寬 1280px
- 1024-1279px:左導覽強制收合為 icon-only
- 1024px 以下:登入頁顯示「請使用桌機瀏覽器」訊息,但仍可繼續使用(畫面以最小寬橫向捲動)
- **不做手機 portrait 排版**

### 無障礙(以 WCAG AA 為目標)

- 文字對比:正常文字 4.5:1、大字 3:1
- Focus ring:2px solid `--color-focus`,offset 2px,所有可互動元件必有
- Tab 順序符合視覺順序;表格列、按鈕、輸入皆可 focus
- Esc 關閉 modal / drawer
- 必填欄位 label 含「(必填)」文字(螢幕閱讀器讀得到),不只靠紅點
- 圖示按鈕必有 `aria-label`
- 狀態不可僅靠顏色;一定搭配 icon 或文字

## 34. 與 SPEC.md 對應與衝突處理

- 顏色 / 字級 / 間距 / 圓角採 Figma Variables,**變數名稱對齊本檔 token 名**(`color-primary`、`status-success`、`fs-2`、`space-4` 等)

### 命名規範

- 元件層級用 kebab-case 英文:`button/primary/default`、`badge/status/pending`
- 圖層內中文僅限畫面文字本身
- Icon 用 Lucide 原名:`Plus`、`Pencil`、`Trash2`,方便切版直接引用

### 審稿流程

1. **第一輪**:Lina + Sophia + Tom 一起看畫稿、給意見(Figma 留言)
2. **第二輪**:依意見修改後,給 Verna(餐飲部 viewer 視角)做使用情境試讀
3. **第三輪**:技術可行性對齊(由開發方確認每個元件可實作、token 對得上、icon 可引用)
4. 每輪結束後將決議回寫到本檔對應章節

### 切版交接檢查

開發者拿到 Figma 後,需檢查:

- 所有顏色 / 字級 / 間距 / 圓角是否皆引用 token,未寫死
- 所有 icon 是否為 Lucide 名稱
- 互動狀態是否完整(8 種狀態見 §35 第 12 點)
- 是否有跨頁元件不一致(例如同個 badge 在兩頁用不同色)

任一項缺失,設計需補完才能進入切版。
